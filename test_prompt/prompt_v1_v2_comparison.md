# Prompt v1 / v2 更改对照

这份文档用于说明：任务9中的 `prompt_v2.py` 是如何基于任务8评估结果，从任务7同学的原始 Prompt 迭代而来的。

## 文件对应

- v1 来源：
  [task7_test.py](/mnt/d/ayibike/video/education/second/EducationAgent/test_prompt/task7_test.py:13)
- v2 来源：
  [prompt_v2.py](/mnt/d/ayibike/video/education/second/EducationAgent/test_prompt/prompt_v2.py:1)

## 总体变化

v1 的核心思路是“尽量规范输出”，已经解决了基础格式、类型枚举、关系枚举的问题。  
v2 的核心思路是“减少误抽和过度解读”，重点收紧实体准入、关系触发条件和类型边界。

简化理解：

- v1 更像“请抽取知识图谱，并按格式输出”
- v2 更像“宁可少抽，也不要抽错、不要过度解释”

## 一、修改目标对照

| 任务8发现的问题 | v1 情况 | v2 修改方向 |
|---|---|---|
| 评测维度误作独立节点 | 没有明确禁止 | 显式禁止把评测指标、评价维度、性质、子属性抽成实体 |
| 关系方向错误 | 只列了关系名，没有足够方向约束 | 单独加入 `包含` 的方向规则和校验 |
| 定义空洞或定义混淆 | 只要求“简短定义”，约束不够强 | 要求写核心区分特征，信息不足可留空 |
| 实体类型标注错误 | 类型定义较粗 | 增加 `concept / algorithm / tool` 的边界说明和反例 |
| 冗余/过于工具化的实体 | 只禁止表格列名、数值、人名地名 | 增加“过于具体、过于工具化、局部细节不提取” |
| 关系过度概括/过度解读/冗余 | 只规定关系类型，没有限制抽取密度 | 明确要求“关系要少而精”，只保留原文直接支持的关系 |
| 跨 chunk 重复 | Prompt 无法解决 | 仍不在 Prompt 中处理，留给任务11 |

## 二、逐项更改对照

### 1. 总体目标从“抽取”改为“少而准”

**v1：**

```text
你是一个专业的知识图谱构建助手，擅长从教材中提取结构化知识。
请从以下教材段落中提取知识图谱
```

**v2：**

```text
你是一个专业的知识图谱构建助手，擅长从教材中提取结构化知识。
你的目标不是“尽量多提取”，而是只保留对构建课程知识图谱有价值、且能被原文直接支持的实体与关系。
```

**修改原因：**

- 任务8评估和人工反馈都指出：第一版不是完全抽错，而是“抽得太多、太泛、太满”
- 所以 v2 先把模型目标改成“宁缺勿滥”

### 2. 实体抽取规则被收紧

**v1：**

```text
1. 只提取该段落中明确出现的概念，不要推断或补充段落没有提到的内容
4. 如果多个词是同义词，只保留最常用的那一个实体，其余不要提取
5. 不要提取书中举例用的具体数据（如表格列名、数值、人名、地名）
```

**v2 新增或强化：**

```text
只提取段落中明确出现、且对理解该段内容有稳定价值的概念。
不要把评测指标、评价维度、优缺点、性质、子属性单独提取为实体。
如果一个词只是过于具体、过于工具化、只在本段局部出现且不适合作为长期复用节点，也不要提取。
如果拿不准某个词是否应成为实体，宁可不提取。
```

**修改原因：**

- 对应任务8中的：
  - 评测维度误作节点
  - 冗余/过于工具化的实体
  - 过度概括导致的边界失控

### 3. 新增“评测维度不能抽实体”的硬规则

**v1：**

- 没有这条规则

**v2：**

```text
不要把某个概念的评测指标、评价维度、优缺点、性质、子属性单独提取为实体，除非该段明确把它定义为一个独立概念。
```

**修改原因：**

- 直接对应任务8中的典型错误：
  - 客观性
  - 简单易懂性
  - 互洽性
  - 计算效率

这些更像“评价维度”，不是知识图谱中的稳定概念节点。

### 4. 类型边界从“列举”变成“带反例说明”

**v1：**

```text
实体类型（type）只能是以下五种之一：
- concept
- algorithm
- task
- process
- tool
```

**v2：**

```text
实体类型（type）只能是以下五种之一：
- concept   核心概念、统计量、图表、数据表示、理论对象
- algorithm 算法、模型、方法、检验、估计方法
- task      任务、目标、应用场景
- process   流程、阶段、步骤
- tool      软件、框架、库、平台、系统工具

可视化图表、统计图形、表格表示通常标为 concept，不是 tool。
方法、模型、检验、系数、抽样方法通常标为 algorithm，不是 tool。
只有明确的软件、框架、库、平台才标为 tool。
如果拿不准类型，优先标为 concept。
```

**修改原因：**

- 直接对应任务8中的类型错误：
  - Boxplots 被标成 algorithm
  - 散点图被标成 tool
  - 列联表被标成 tool

### 5. definition 要求明显收紧

**v1：**

```text
definition: 基于原文内容的简短定义，不超过50字，不能只是翻译实体名称本身
```

**v2：**

```text
definition 必须基于原文内容，用一句简短中文说明该实体的核心区分特征。
definition 不能只是翻译实体名称，也不能写成“与XX相关的概念”“一种方法”这类空泛描述。
如果原文对该实体没有提供足够定义信息，可以写空字符串 ""，不要编造。
```

**修改原因：**

- 对应任务8中的：
  - 定义空洞
  - 定义有误
  - 不同实体写出同样定义

这条的核心不是让定义更长，而是避免“硬编定义”。

### 6. 关系抽取从“可抽”变成“严格限抽”

**v1：**

- 给出了 6 类关系，但没有明确限制关系密度

**v2：**

```text
只抽取原文中有直接文本依据的关系；如果需要较强推理、跨句补全或常识脑补，则不要抽取。
关系要少而精，只保留最核心、最明确、最不冗余的关系；不要把同段中所有实体都互相连一遍。
如果关系只是弱相关、泛相关、背景相关，不要输出。
如果没有明确关系，relations 可以为空数组。
```

**修改原因：**

- 对应你补充的评估同学反馈：
  - 存在很多过度概括和过度解读的关系
  - 关系逻辑上可能不完全错，但太冗余

v2 的处理策略不是“换一种关系名”，而是直接减少关系触发。

### 7. `包含` 关系方向被写成硬规则

**v1：**

```text
包含 A 包含 B（上下位关系）
```

**v2：**

```text
包含 source 是上位类/总体，target 是下位类/组成部分

“A 是一种 B” => source=B, target=A，关系填“包含”
“A 包括 B” => source=A, target=B，关系填“包含”
若你无法明确判断方向，就不要输出这条关系
```

**修改原因：**

- 直接对应任务8中的关系方向错误：
  - `binary variable → categorical data`
  - `indicator variable → binary variable`

v2 不再只说“上下位关系”，而是把映射规则写死。

### 8. 同义词和对比关系进一步拆开

**v1：**

```text
如果多个词是同义词，只保留最常用的那一个实体
对比是同类方法的比较（注意：同义词不是对比）
```

**v2：**

```text
如果多个词在本段中是同义词、近义表述或别名，只保留一个最常用、最核心、最适合作为节点的表述；不要把同义词之间标成“对比”。
同义词、别名、翻译对应词，不属于“对比”
```

**修改原因：**

- v1 虽然提到“同义词不是对比”，但强度不够
- v2 把这条拆进实体规则和关系规则两边，避免模型在两个位置漏判

### 9. 输出前增加自检清单

**v1：**

- 没有明确的输出前自检步骤

**v2：**

```text
输出前请自行检查：
- 每个实体名都在原文中出现
- 每个 definition 不是空泛套话
- 每条关系都能被原文直接支持
- “包含”关系方向是否正确
- 是否误把评测指标、属性、例子抽成实体
```

**修改原因：**

- 这是为了把任务8中的高频错误，在生成前再拦一道
- 不保证完全消除，但能显著降低明显错误

## 三、哪些问题没有放进 v2 Prompt

以下问题没有被强行塞进 Prompt，因为它们本质上不是单 chunk Prompt 能稳定解决的：

### 1. 跨 chunk 重复

例如：

- `histogram`
- `random sampling`
- `boxplot`

这些重复是因为每个 chunk 独立抽取。  
这类问题应在任务11的“实体归一化/合并”阶段处理，而不是继续把 Prompt 写得越来越复杂。

### 2. 多书之间的别名合并

例如：

- `机器学习`
- `ML`
- `machine learning`

这属于全局归一化问题，不是任务9当前要靠单次 Prompt 解决的。

## 四、结论

v2 不是对 v1 做“格式微调”，而是围绕任务8的评估结果，做了三类方向性收紧：

1. 实体更严格
- 少抽冗余、少抽属性、少抽评测维度

2. 关系更保守
- 减少过度概括、过度解读和弱关系

3. 类型和方向更明确
- 减少 `concept / tool / algorithm` 误判
- 减少 `包含` 方向错误

如果后续复跑结果显示“关系仍然过多”，下一轮 v3 可以继续只针对“关系触发条件”再收紧，而不用大改整个 Prompt。
