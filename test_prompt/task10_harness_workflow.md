# 任务10 Harness Workflow

任务10不是“一次性把 10 本书全跑完”就结束，而是继续沿用任务9的 harness 思维：

1. 跑一轮批量抽取
2. 抽样复查结果
3. 记录 prompt 问题
4. 修改 prompt
5. 再跑一轮
6. 比较新旧结果
7. 保留最优版本，继续扩大规模

也就是说，任务10仍然是一个“批量抽取实验流水线”，不是单纯脚本执行。

## 角色分工

建议至少按这 3 个角色来拆：

### 1. 抽取执行者
- 负责运行批量抽取脚本
- 维护 batch-name、summary、manifest
- 保证结果持续落盘、可续跑

### 2. 结果批判者
- 负责抽样查看批量结果
- 专门指出：
  - prompt 漏掉了什么
  - prompt 多抽了什么
  - 什么关系在胡连
  - 哪些定义开始漂了

### 3. Prompt 修改者
- 根据批判意见改 prompt
- 不能凭感觉乱改
- 每次修改必须说明：
  - 改了哪条规则
  - 对应哪类错误
  - 为什么这样改

## 每一轮必须产出什么

每一轮都必须在 `test_prompt/task10_rounds/` 下留下记录，至少包括：

1. 本轮 batch 运行信息
- 用了哪个 prompt 版本
- 跑了哪几本书
- 处理了多少 chunk
- 成功多少，失败多少

2. 本轮主要问题
- 高频问题有哪些
- 是实体问题还是关系问题
- 是 prompt 问题还是后处理问题

3. 本轮 prompt 修改
- 改动前是什么
- 改动后是什么
- 预期改善什么

4. 本轮结论
- 是否继续保留当前 prompt
- 是否需要下一轮
- 是否可以扩大批量规模

## 什么时候该改 prompt，什么时候不该改

### 应该改 prompt
- 漏掉本段核心概念
- 把评测维度、属性、例子误抽成节点
- 同义词重复
- 关系类型误判
- 关系方向错误
- 定义明显空泛

### 不应该继续靠 prompt 硬改
- 跨 chunk 重复
- 跨书别名归一
- 全局合并
- 节点标准化

这些应留给任务11。

## 推荐执行节奏

### 阶段 A：小规模批量
- 2 本书
- 每本 5-20 个 chunk
- 目的：验证 prompt 在“连续多段”上的稳定性

### 阶段 B：中规模批量
- 前 10 本书
- 目的：验证跨书稳定性

### 阶段 C：当前全部可用书目
- 当前 15 本
- 目的：形成任务10正式产物

每扩大一个阶段，都必须先留一轮 round 记录。
